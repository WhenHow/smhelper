# 自媒体运营助手：直播助手第一阶段 PRD

## 1. 背景与目标

本项目是一个自媒体运营助手，长期目标是通过系统化管理自媒体平台账号、直播任务、产品资料和提问策略，辅助运营在直播间提出高质量问题，引导主播围绕产品卖点进行回答。

第一阶段聚焦小红书直播助手闭环：中心节点打开直播间并提取拉流地址，持续录制直播内容；系统转写音频并生成候选问题；运营在后台管理页面审核、编辑并批准发送；系统调度远端操作节点使用自有授权账号进入直播间等待并发送问题。

第一阶段要验证的核心闭环是：

1. 运营创建小红书直播任务，配置直播间 URL、产品资料和可用账号。
2. 中心匿名 observer 打开直播间 URL，判断直播进行中并提取 `.flv` / `.m3u8` 拉流地址。
3. 中心使用长期 ffmpeg 进程固定时长切片录制直播视频。
4. 下一个 segment 出现时，系统认为前一个 segment 完成，并异步截取首尾帧、提取音频。
5. 音频提交 ASR 厂商服务转写为文本。
6. 系统使用 LiteLLM 调用 LLM，结合产品资料、近期转写文本和任务上下文生成 1 个 JSON 化候选问题；首尾帧截图仍需生成和保存，但第一阶段不作为 LLM 输入。
7. 运营通过 SQLAdmin 后台审核、编辑候选问题并批准发送。
8. 系统选择符合规则的已入场账号会话，通过远端操作节点在小红书直播间发送问题。
9. 系统记录任务状态、处理产物、错误原因、账号会话状态和发送日志。

## 2. 阶段范围

### 2.1 本阶段包含

- 支持小红书直播助手端到端流程。
- 支持 FastAPI + SQLAdmin 后台，用于候选问题审核、编辑、批准发送和基础数据查看。
- 支持单管理员后台登录，不做复杂角色权限。
- 支持中心匿名 observer 访问直播间 URL，提取直播流地址并持续判断直播状态。
- 支持 `ffmpeg` 长期进程固定时长 segment 录制、首尾帧截图和音频提取。
- 支持 ASR 厂商 adapter 转写，应用层通过统一接口调用。
- 支持 LiteLLM 生成 JSON 化候选问题，并默认设置 `LITELLM_LOCAL_MODEL_COST_MAP=True`，避免运行时拉取 GitHub 模型价格数据。
- 支持全量可用账号按 HRW/Rendezvous Hashing 分配到远端操作节点。
- 支持账号按 15-45 秒随机间隔逐个进入直播间等待。
- 支持远端 Celery worker 节点保持账号浏览器会话，并在收到发送任务后留言。
- 支持 MySQL 持久化任务状态、处理产物元数据、候选问题、审核记录、账号会话、发送记录和错误日志。
- 支持本地文件保存视频片段、音频文件、截图和第一阶段明文 `storage_state.json`。
- 支持 Celery + Redis 执行中心异步任务和远端浏览器操作任务。
- 支持账号发送冷却、发送上限和审计。

### 2.2 本阶段不包含

- 完整定制 Web 管理后台；第一阶段先使用 SQLAdmin。
- CLI 人工审核编辑候选问题。
- 多平台真实落地实现。
- 自动发送闭环；所有问题必须人工批准。
- 复杂角色权限。
- Worker 节点鉴权；第一阶段假设可信内网或开发环境。
- 登录态加密；第一阶段为观察允许中心明文保存，但不得提交仓库或打印日志。
- 文档上传、知识库检索或 RAG。
- ASR 多供应商自动 fallback。
- 任务级 LLM/ASR 供应商选择。
- 资源指标监控、Worker 运维面板或完整运营 SOP。

## 3. 用户与使用场景

### 3.1 主要用户

- 运营人员：配置直播任务、观察任务状态、审核候选问题、编辑最终发送文本、确认发送。
- 管理人员：维护自有授权账号、产品资料、AI 供应商配置、节点配置和任务规则。

第一阶段采用单管理员模式，不区分复杂角色权限。

### 3.2 典型场景

运营人员预先准备一场小红书直播任务，录入直播间 URL、产品资料和可用账号。直播开始后，中心 observer 确认直播状态并提取拉流地址，ffmpeg 持续录制；系统自动处理片段、转写音频并生成候选问题。所有可用账号按节点分配策略陆续进入直播间等待。运营在 SQLAdmin 后台查看候选问题，编辑确认后触发发送。系统从等待中的账号会话里选择合适账号发送，并记录完整日志。

## 4. 核心业务流程

### 4.1 任务准备

任务启动前需要准备以下信息：

- 小红书直播任务基本信息。
- 小红书直播间 URL。
- 产品结构化资料，包括产品卖点、禁问点、目标人群、话术要求和其他生成约束。
- 可用的自有授权账号及其 `storage_state.json` 登录态。
- 账号允许节点集合。
- WorkerNode 配置，包括节点 ID、Celery 队列名、支持平台和 `max_browser_sessions`。
- 全局默认 ASR 供应商配置。
- LiteLLM LLM 配置，包括主模型和可选 fallback。
- 录制 segment 时长、账号入场间隔、账号冷却、发送上限等运行配置。

第一阶段不要求提供完整业务 CRUD 体验，但基础对象可以通过 SQLAdmin 管理。

### 4.2 中心直播观察与流地址发现

任务启动后，中心节点创建匿名 observer 页面：

1. 打开直播间 URL。
2. 等待直播状态信号，例如播放器、视频元素、拉流请求或直播结束提示。
3. 如果直播未开始或已结束，任务进入对应状态。
4. 如果直播进行中，捕获 `.flv` / `.m3u8` 拉流地址。
5. 保存 stream URL、发现时间、发现来源和页面状态。
6. 保持 observer 页面打开，持续判断直播状态。

stream URL 是运行时产物，不要求人工配置，也不应长期复用。

### 4.3 直播采集

中心节点使用 observer 提取到的 stream URL 启动长期 ffmpeg 进程：

1. ffmpeg 使用固定 `segment_time_seconds` 连续输出视频片段，第一阶段默认 60 秒。
2. 正常运行中，系统以下一个 segment 出现作为前一个 segment 完成信号。
3. LiveTask 停止或 ffmpeg 退出时，对最后一个 segment 做一次收尾确认。
4. 完成的 segment 保存到本地媒体目录。
5. 系统从 segment 中截取首帧和尾帧；截图用于采集留存、监控审计和后续扩展，第一阶段不作为 LLM 上下文。
6. 系统从 segment 中提取音频文件。
7. 数据库记录视频、截图、音频的文件路径、生成时间、处理状态和错误信息。

录制链路不加人为抖动。ASR 和 LLM 在产物完成后尽快异步执行。

### 4.4 ASR 转写

ASR 使用厂商能力，不走 LiteLLM。系统通过 `SpeechToTextService` 接口调用具体 provider adapter。

音频文件生成后：

- 提交音频给当前配置的 ASR 厂商。
- adapter 内部封装同步/异步厂商差异。
- 获取转写文本。
- 保存转写结果和供应商响应摘要。
- 失败时记录错误原因，并允许后续人工或任务重试。

### 4.5 LLM 生成候选问题

系统使用 LiteLLM Python SDK 调用 LLM。运行环境必须设置：

```text
LITELLM_LOCAL_MODEL_COST_MAP=True
```

系统将以下内容提交给 LLM：

- 产品结构化资料。
- 当前直播任务基本信息。
- 当前 segment 的 ASR 转写文本，可附带近期上下文摘要。
- 已生成或已发送的问题摘要，用于避免重复。
- 生成约束，例如禁问点、问题语气和产品相关性要求。

每个完成 ASR 的 segment 默认生成 1 个候选问题。LLM 必须返回 JSON：

```json
{
  "question": "候选问题文本",
  "reason": "生成理由",
  "risk_level": "low"
}
```

系统保存 `question`、`reason`、`risk_level`、`raw_response` 和解析状态。解析失败时记录失败产物，不进入待审核队列。

### 4.6 账号全量入场等待

中心 observer 确认直播进行中并提取 stream URL 后，中心并行启动录制链路和账号入场链路。

账号入场规则：

1. 筛选所有可用账号。
2. 对每个账号，根据账号允许节点集合和 HRW/Rendezvous Hashing 选择稳定目标节点。
3. 按 15-45 秒随机间隔逐个向目标节点 Celery queue 投递入场任务。
4. 远端节点通过中心 API 获取该账号的 `storage_state.json`。
5. 远端节点打开 CloakBrowser，进入直播间，必要时取消静音。
6. 入场成功后创建或更新 `AccountLiveSession` 为 `waiting`。

所有可用账号都应尝试进入直播间等待。节点容量通过 `max_browser_sessions` 配置管理，节点应按账号规模规划容量。

同一个 `account_id + live_task_id` 同一时间只能有一个 active `AccountLiveSession`，避免同一账号打开多个浏览器进入同一直播间。active 状态包括 `planned`、`starting`、`waiting`、`sending`、`closing`。如果 session 异常，系统可以自动重建，但必须先终结旧 session；每个账号每场直播最多自动重建 2 次。

直播结束后，中心进入收场流程：停止新增候选问题、停止创建入场任务和发送任务，向所有 active sessions 投递关闭指令。正在发送的 session 最多等待 30 秒；超时后关闭并记录 `shutdown_timeout`。节点无响应时，中心将相关 session 标记为 `lost`。

### 4.7 后台审核与发送

候选问题默认进入待审核状态。运营人员通过 SQLAdmin 后台查看候选问题，可以编辑最终发送文本。

批准发送时：

1. 系统保存 LLM 原始候选文本和运营最终发送文本。
2. 创建 `DispatchJob`。
3. 从当前 LiveTask 的 `waiting` sessions 中选择发送账号。
4. 选择策略为带权随机，偏向近期发送较少、空闲更久的账号，但避免严格轮转。
5. 同一 LiveTask 允许多个账号并发发送。
6. 同一账号/同一 session 同一时刻只能执行一个发送动作。
7. 远端节点在已有直播间页面填入文本并点击发送。
8. 第一阶段只要页面操作完成且未抛异常，即记录发送成功。
9. 只有发送成功才进入账号冷却。
10. 发送失败不自动重试，只记录失败原因和页面状态。

## 5. 后台管理能力

第一阶段后台采用 FastAPI + SQLAdmin，不自研完整审核 UI。

### 5.1 登录

- 使用单管理员登录。
- 管理员账号密码从配置或环境变量读取。
- 不做复杂角色权限。

### 5.2 LiveTask 管理

至少展示：

- 任务名称、平台、直播间 URL。
- 当前任务状态。
- observer 状态。
- 当前 stream URL 摘要。
- 最近 segment。
- 最近错误原因。

### 5.3 CandidateQuestion 审核

至少支持：

- 查看候选问题文本、生成理由、风险等级、关联 segment、ASR 摘要。
- 编辑最终发送文本。
- 批准发送。
- 拒绝或忽略。
- 查看发送状态。

### 5.4 AccountLiveSession 查看

至少展示：

- 账号 ID。
- 所属 LiveTask。
- 所属节点。
- session 状态。
- 最近心跳。
- 最近发送时间。
- 失败原因。

### 5.5 登录态管理边界

SQLAdmin 不直接展示或编辑完整 `storage_state.json`。后台只展示：

- 账号 ID。
- 平台。
- 登录态状态。
- 存储路径。
- 最近验证时间。
- 更新时间。

真实登录态内容不得在列表页、详情页或日志中直接输出。

## 6. 数据与任务边界

### 6.1 数据存储

- 业务数据使用 MySQL。
- 媒体文件保存在本地文件目录。
- 第一阶段登录态可明文保存在中心本地 `data/` 或受控路径中，方便观察。
- `data/` 下真实数据不应提交仓库。
- 数据库保存文件路径、元数据、处理状态和错误信息。

### 6.2 Celery 任务

中心侧任务包括：

- observer 启动与流地址发现。
- ffmpeg 录制管理。
- segment 完成扫描。
- 首尾帧截图。
- 音频提取。
- ASR 转写。
- LLM 候选问题生成。
- DispatchJob 创建和发送调度。

远端节点任务包括：

- 账号进入直播间。
- session 健康检查。
- 发送留言。
- 关闭 session。

Celery payload 不携带完整 `storage_state.json`。节点收到任务后通过中心 API 拉取登录态，执行结果通过中心 API 回传。

### 6.3 平台边界

第一阶段只实现小红书真实链路，但代码结构应保留平台适配边界：

- 通用流程不应直接散落小红书页面选择器、发送细节或登录态细节。
- 小红书相关能力应集中在 `platforms/xhs`。
- CloakBrowser 通用能力放在 `infrastructure/browser/cloakbrowser`。
- 后续扩展其他平台时，应复用任务、账号、采集、ASR、LLM、审核和日志主流程。

## 7. 风控与审计

第一阶段默认采用强限制和审计。

必须支持：

- 账号冷却时间。
- 账号每日发送上限。
- 单账号同一时刻只允许一个发送动作。
- 同一个账号在同一 LiveTask 同一时间只能有一个 active session。
- 异常 session 自动重建前必须先终结旧 session，每账号每场直播最多重建 2 次。
- 直播结束后中心负责关闭所有 active sessions，发送中的 session 最多等待 30 秒。
- 账号入场 15-45 秒随机间隔。
- 带权随机选择发送账号，避免严格顺序轮转。
- 黑名单词检查。
- 人工审核默认开启。
- 自动发送默认关闭。
- 账号使用日志。
- 发送成功和失败日志。
- 失败原因记录。
- 发送成功判定方式记录为 `operation_completed`。

系统不得设计绕过平台访问控制、验证码、风控策略或其他限制的能力。

## 8. 验收标准

第一阶段完成后，应满足以下验收标准：

- SQLAdmin 后台可登录并查看直播任务、候选问题、账号会话和发送记录。
- 中心 observer 能匿名打开小红书直播间 URL，判断直播状态并提取拉流地址。
- ffmpeg 能使用提取到的 stream URL 长期固定时长切片。
- 下一个 segment 出现时，前一个 segment 能进入后处理。
- 单个 segment 能完整产出视频、首帧、尾帧、音频、ASR 文本和 1 个 JSON 化候选问题。
- 第一阶段 LLM 候选问题不依赖图片输入。
- 候选问题默认不会自动发送。
- 运营能在 SQLAdmin 中编辑候选问题并批准发送。
- 所有可用账号能按 HRW 分配到操作节点，并以 15-45 秒随机间隔陆续进入直播间等待。
- 同一账号不会在同一直播任务中重复创建多个 active 浏览器会话。
- 异常账号会话可以在关闭旧会话后自动重建，且重建次数受限。
- 运营批准后，系统能使用一个 waiting session 尝试发送最终文本。
- 同一 LiveTask 允许多个账号并发发送，但同一账号/session 不并发发送。
- 直播结束后，中心能关闭或标记所有账号会话，LiveTask 不会因无响应 worker 永久卡住。
- 发送失败不自动重试，只记录失败原因。
- 所有关键动作都有持久化日志。
- 账号调度遵守冷却、上限、平台、可用状态和节点容量规则。
- 媒体文件保存在本地目录，数据库可追踪其路径和处理状态。
