# 自媒体运营助手：直播助手第一阶段 PRD

## 1. 背景与目标

本项目是一个自媒体运营助手，长期目标是通过系统化管理自媒体平台账号、直播任务、产品资料和提问策略，辅助运营在直播间提出高质量问题，引导主播围绕产品卖点进行回答。

第一阶段聚焦“直播助手”的 CLI MVP：先通过命令行跑通小红书直播间的完整链路，并具备基础监控和控制能力。Web 管理后台仍是产品方向，但不作为第一阶段闭环验收前置。

第一阶段要验证的核心闭环是：

1. 读取已配置的小红书直播任务和直播流地址。
2. 使用 `ffmpeg` 按时间窗口录制直播视频片段。
3. 从视频片段中截取首帧、尾帧，并提取音频。
4. 将音频提交 ASR 服务转写为文本。
5. 将产品资料、近期转写文本和任务上下文提交给 LLM；首尾帧截图仍需生成和保存，但第一阶段不作为 LLM 输入。
6. LLM 生成候选问题。
7. 运营人员通过 CLI 人工确认候选问题。
8. 系统选择符合规则的自有授权账号，通过 CloakBrowser 进入直播间并发送问题。
9. 系统记录任务状态、处理产物、错误原因和发送日志。

## 2. 阶段范围

### 2.1 本阶段包含

- 支持小红书直播助手端到端流程。
- 支持 CLI 启动、查看、停止直播任务，查看日志，并人工确认候选问题。
- 支持人工配置直播流地址。
- 支持 `ffmpeg` 视频录制、首尾帧截图和音频提取。
- 支持 ASR 转写和 LLM 候选问题生成。
- 支持自有授权账号进入直播间并发送人工确认后的问题。
- 支持 MySQL 持久化任务状态、处理产物元数据、候选问题、审核记录、发送记录和错误日志。
- 支持本地文件保存视频片段、音频文件和截图。
- 支持 Celery + Redis 执行异步任务。
- 支持账号发送强限制和审计。

### 2.2 本阶段不包含

- 完整 Web 管理后台。
- 多平台真实落地实现。
- 自动发送闭环。
- 浏览器网络请求解析直播流地址。
- 文档上传、知识库检索或 RAG。
- 多供应商自动容灾切换。
- 任务级供应商选择。
- 复杂角色权限。
- 资源指标监控、Worker 运维面板或完整运营 SOP。

## 3. 用户与使用场景

### 3.1 主要用户

- 运营人员：配置直播任务、观察任务状态、审核候选问题、确认发送。
- 管理人员：维护自有授权账号、产品资料、AI 供应商配置和任务规则。

第一阶段暂不区分复杂角色权限，可按单管理员模式设计。

### 3.2 典型场景

运营人员预先准备一场小红书直播任务，录入直播流地址、产品资料和可用账号。直播开始后，运营通过 CLI 启动任务。系统自动分段采集直播内容，并根据直播上下文生成候选问题。运营查看候选问题后，选择合适的问题确认发送。系统使用符合规则的账号进入直播间提问，并记录完整日志。

## 4. 核心业务流程

### 4.1 任务准备

任务启动前需要准备以下信息：

- 小红书直播任务基本信息。
- 直播间访问信息和人工配置的直播流地址。
- 产品结构化资料，包括产品卖点、禁问点、目标人群、话术要求和其他生成约束。
- 可用的自有授权账号。
- 全局默认 ASR 供应商配置。
- 全局默认 LLM 供应商配置。
- 录制窗口、上下文窗口、账号冷却、发送上限和随机间隔等运行配置。

第一阶段不要求 CLI 提供完整 CRUD。以上配置可以先通过配置文件、数据库初始化或后续后台维护。

### 4.2 直播采集

任务启动后，系统按配置周期执行直播切片：

1. 使用 `ffmpeg` 根据直播流地址录制指定时长的视频片段。
2. 视频片段保存到本地媒体目录。
3. 系统从视频片段中截取首帧和尾帧；截图用于采集留存、监控审计和后续扩展，第一阶段不作为 LLM 上下文。
4. 系统从视频片段中提取音频文件。
5. 数据库记录视频、截图、音频的文件路径、生成时间、处理状态和错误信息。

### 4.3 ASR 转写

音频文件生成后，系统调用当前启用的 ASR 供应商：

- 上传或提交音频文件。
- 获取转写文本。
- 保存转写结果和供应商响应摘要。
- 失败时记录错误原因，并允许后续人工或任务重试。

系统应维护最近若干段转写文本作为上下文窗口，供 LLM 生成问题时使用。

### 4.4 LLM 生成候选问题

系统将以下内容提交给当前启用的 LLM 供应商：

- 产品结构化资料。
- 当前直播任务基本信息。
- 最近若干段 ASR 转写文本。
- 已生成或已发送的问题摘要，用于避免重复。
- 生成约束，例如禁问点、问题语气和产品相关性要求。

第一阶段暂不将首帧、尾帧或其他截图作为 LLM 输入。截图仍作为采集产物保存，供监控、审计和后续多模态能力扩展使用。

LLM 返回候选问题后，系统保存候选问题、生成来源、关联切片、状态和错误信息。

### 4.5 人工审核与发送

候选问题默认进入待审核状态。运营人员通过 CLI 查看候选问题，并选择确认发送。

发送前系统必须检查：

- 账号是否可用。
- 账号是否属于当前平台。
- 账号是否处于冷却期。
- 账号是否超过每日发送上限。
- 当前任务是否超过账号并发或发送上限。
- 问题是否包含黑名单词或明显违反生成约束的内容。

通过检查后，系统选择符合规则的账号，通过 CloakBrowser 使用该账号进入小红书直播间并发送问题。发送结果、失败原因和账号使用记录必须写入日志。

## 5. CLI 能力

第一阶段 CLI 聚焦直播任务生命周期，不承担完整后台 CRUD。

### 5.1 `smhelper live start <task>`

启动指定直播任务。

应完成：

- 读取任务配置。
- 校验直播流地址、产品资料、账号池和 AI 供应商配置。
- 创建任务运行记录。
- 启动或调度后续采集任务。

### 5.2 `smhelper live status <task>`

展示任务当前状态。

至少展示：

- 当前任务阶段。
- 最近切片信息。
- 视频、音频、首帧、尾帧产物状态。
- ASR 状态。
- LLM 状态。
- 候选问题状态。
- 最近发送状态。
- 最近错误原因。

### 5.3 `smhelper live logs <task>`

查看任务运行日志。

日志至少覆盖：

- 启动。
- 录制。
- 截图。
- 音频提取。
- ASR。
- LLM。
- 候选问题生成。
- 人工审核。
- 发送。
- 失败。
- 停止。

### 5.4 `smhelper live approve <candidate>`

人工确认候选问题并触发发送。

应完成：

- 校验候选问题状态。
- 执行账号选择和发送限制检查。
- 使用 CloakBrowser 发送问题。
- 更新候选问题、发送记录和账号使用记录。

### 5.5 `smhelper live stop <task>`

停止指定直播任务。

应完成：

- 标记任务为停止中或已停止。
- 终止后续切片调度。
- 避免继续生成新问题或发送新问题。
- 记录停止时间和停止原因。

## 6. 数据与任务边界

### 6.1 数据存储

- 业务数据使用 MySQL。
- 媒体文件保存在本地文件目录。
- 数据库只保存文件路径、元数据、处理状态和错误信息。
- 后续 Web 后台应复用第一阶段 CLI 产生的数据模型和日志。

### 6.2 异步任务

使用 Celery + Redis 编排后台任务。

核心任务类型包括：

- 直播切片录制。
- 首尾帧截图。
- 音频提取。
- ASR 转写。
- LLM 候选问题生成。
- 审核后发送。
- 失败重试或人工重试。

任务应尽量保持幂等。重复执行同一处理步骤时，不应产生不可控的重复发送。

### 6.3 平台边界

第一阶段只实现小红书真实链路，但代码结构应保留平台适配边界：

- 通用流程不应直接散落小红书页面选择器、发送细节或登录态细节。
- 小红书相关能力应集中在平台适配层。
- 后续扩展其他平台时，应复用任务、账号、采集、ASR、LLM、审核和日志主流程。

## 7. 风控与审计

第一阶段默认采用强限制和审计。

必须支持：

- 账号冷却时间。
- 账号每日发送上限。
- 单任务发送上限。
- 随机发送间隔。
- 黑名单词检查。
- 人工审核默认开启。
- 自动发送默认关闭。
- 账号使用日志。
- 发送成功和失败日志。
- 失败原因记录。

系统不得设计绕过平台访问控制、验证码、风控策略或其他限制的能力。

## 8. 验收标准

第一阶段完成后，应满足以下验收标准：

- CLI 能启动、查看、停止一个直播任务。
- CLI 能展示任务阶段、最近产物、候选问题、发送状态和错误原因。
- 单个直播切片能完整产出视频、首帧、尾帧、音频、ASR 文本和 LLM 候选问题；第一阶段 LLM 候选问题不依赖图片输入。
- 候选问题默认不会自动发送。
- 运营通过 CLI 确认候选问题后，系统能尝试使用符合规则的自有授权账号进入直播间发送。
- 所有关键动作都有持久化日志。
- 账号调度遵守冷却、上限、平台、可用状态和随机间隔规则。
- 媒体文件保存在本地目录，数据库可追踪其路径和处理状态。
- ASR 和 LLM 供应商配置支持多条维护，但每类服务第一阶段只启用一个全局默认供应商。

## 9. 后续方向

第一阶段 CLI 跑通后，后续可以逐步补充：

- Web 管理后台。
- 账号、产品、供应商和任务的后台 CRUD。
- 审核队列页面。
- 任务监控页面。
- 多平台适配。
- 自动发送策略。
- 文档上传和知识库检索。
- 多供应商容灾切换。
- 更完整的角色权限和操作审计。
