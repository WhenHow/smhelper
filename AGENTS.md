## 工作规则

- 默认使用中文回复，除非用户明确要求其他语言。
- 每次回复用户时，必须先说“好的，老板”。
- 修改代码前先理解现有结构；不要主动做与任务无关的重构。
- 实现前必须明确关键假设；如果需求存在多种理解，应先说明分歧点并向用户确认，不要静默选择一种解释。
- 如果发现更简单、更稳妥的方案，应主动说明取舍；当需求可能导致过度设计或风险较高时，应提出反对意见和替代方案。
- 涉及架构、依赖、数据库 schema、CLI 行为、任务队列或浏览器自动化的变更，先说明方案并等待确认。
- 对多步骤任务，开始前应给出简短计划，并为每一步说明对应验证方式。
- 完成代码修改前必须运行完整验证；不能在未验证时声称完成。

## 项目需求与功能

- 项目是自媒体运营助手，用于管理自媒体平台账号、直播任务、产品资料和直播间提问流程。
- 第一阶段聚焦“直播助手”的 CLI MVP，先跑通小红书直播间端到端流程，并具备基础监控和控制能力。
- 第一阶段核心链路：人工配置小红书直播流地址，使用 `ffmpeg` 分段录制直播视频，截取首尾帧并提取音频，音频经 ASR 转写后，结合产品资料、近期转写文本和任务上下文提交给 LLM 生成候选问题；首尾帧截图仍需生成和保存，但第一阶段不作为 LLM 输入。
- 候选问题默认由运营人员通过 CLI 人工确认；确认后系统选择符合规则的自有授权账号，通过 CloakBrowser 进入直播间发送问题，并记录任务状态、处理产物、错误原因和发送日志。
- 第一阶段 CLI 应覆盖直播任务启动、状态查看、日志查看、停止和人工确认发送。Web 管理后台是后续方向，不作为第一阶段闭环验收前置。
- 更完整的需求说明见 `docs/live-assistant-prd.md`。


## 项目约定

- 使用 DDD 作为长期设计方向：领域规则应与基础设施细节分离。
- 不提前创建臆测性的目录结构；只有真实功能需要时再添加结构。
- 保持 `src/smhelper/__init__.py` 简洁，不把业务实现堆在入口文件里。
- `CODE_MAP.md` 是项目结构索引，由 Codex hooks 自动生成；理解项目结构时应优先查看 `CODE_MAP.md`，再按需读取具体文件。
- 不要手工编辑 `CODE_MAP.md`；如需调整结构索引的扫描范围，请修改 `.codex/code_map_ignore`。
- 覆盖率产物（如 `.coverage`、`.coverage.*`、`htmlcov/`）属于本地生成文件，不应提交，也不应作为理解项目结构的依据。

## 架构与代码质量

- 保持低耦合、高内聚：每个模块、类、函数应有清晰职责，避免让一个文件同时承担入口、业务规则、数据访问、外部服务调用和展示/编排逻辑。
- 遵循 DDD 分层思路：领域规则优先放在领域模型或领域服务中；应用编排、基础设施适配、外部接口访问应与领域规则分离。
- 优先使用能解决当前问题的最小实现；不要添加用户未要求的功能、配置项、扩展点或抽象。
- 每次规划方案、写代码前，先思考是否已有成熟方案或流行库可用；如果有，应优先采用或复用，避免重复制造轮子。
- 单次使用的代码不要提前抽象；只有出现真实重复、复杂度明显上升或边界需要稳定时，才提取抽象。
- 不要为了“灵活性”添加尚未被需求证明的参数、插件机制、策略模式或兼容层。
- 不要把功能集中堆在单个文件中；当文件职责变多、代码明显难以阅读或测试时，应按职责拆分到已有合适模块，必要时先说明拆分方案并等待确认。
- 新增功能时优先复用已有边界和抽象；只有当现有结构无法清晰表达职责时，才新增模块、类或接口。
- 代码应易读、易测、易替换：业务逻辑避免直接依赖具体外部服务、环境变量、数据库、网络请求或框架细节，应通过清晰的接口或适配层隔离。
- 不为了“架构完整”提前创建空目录、空抽象或未来可能用到的层；结构应跟随真实功能自然增长。

## 依赖与命令

- 只使用 `uv` 管理依赖、虚拟环境和命令执行。
- `uv.lock` 必须提交；依赖变化必须通过 `uv` 更新 `pyproject.toml` 和 `uv.lock`。
- 不要手工编辑 `uv.lock`。

常用命令：

```bash
uv sync
uv add
uv run
```

开发工具作为 dev dependencies 管理：

```bash
uv add --dev ruff mypy pytest pytest-cov
```

## 质量门禁

每次代码修改完成前都必须运行：

```bash
uv run ruff format --check .
uv run ruff check .
uv run mypy src
uv run pytest --cov=src/smhelper --cov-report=term-missing --cov-fail-under=90
```

如果命令失败，先修复再结束任务。若缺少工具，用 `uv add --dev ...` 补齐，不要跳过验证。

## 编码与测试

- 可以对本次修改过的文件运行 `uv run ruff format <files>`；不要无差别格式化整个仓库，除非用户明确要求。
- 公共函数、方法和复杂内部 helper 必须添加类型标注。
- 公共模块、类、函数和方法应有清晰的用途说明；复杂逻辑、领域规则、边界条件、失败处理、兼容性处理和外部约束应添加注释，重点说明“为什么这样做”。
- 避免给显而易见的赋值、调用或流程添加机械注释；注释应帮助理解意图、约束和取舍。
- 避免使用 `Any`；不要添加没有说明的 `# type: ignore`。
- 修改公共行为时必须新增或更新测试。
- 修复 bug 时优先用测试或最小复现证明问题存在，再实现修复并验证通过。
- 测试应覆盖正常路径、边界条件和失败路径，避免只为覆盖率写浅测试。
- 默认单元测试不得依赖真实 Redis、真实数据库、浏览器自动化、网络或外部服务；需要真实外部服务的测试标记为 integration。

## 交付习惯

- 保持提交聚焦，推荐使用 `feat:`、`fix:`、`test:`、`docs:`、`refactor:`、`chore:`。
- 修改应保持聚焦：每一处改动都应能追溯到用户当前请求。
- 不主动“顺手优化”相邻代码、注释、格式或结构；发现无关问题时可以说明，但不要擅自修改。
- 只清理由本次改动产生的无用 import、变量、函数或文件；不要删除既有死代码，除非用户明确要求。
- 修改现有代码时优先匹配当前项目风格，即使存在个人偏好的不同写法。
- 重构前后都应保持测试通过；如果无法自动验证，必须说明人工验证方式和剩余风险。
- 最终回复说明修改了哪些文件、运行了哪些验证命令、是否存在剩余风险。
